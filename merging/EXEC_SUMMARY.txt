───────────────────────────────────────────────────────────────────────────────
                        DESIGN DECISIONS & RATIONALE
───────────────────────────────────────────────────────────────────────────────

Q: Why Person 2's HTTPRequest instead of Person 3's Request?
A: Person 3's Request was a stub (author marked it "NOT my task, only for testing").
   Person 2's HTTPRequest is a proper class with getters/setters, lowercased headers,
   and chunked decoding built in. No ambiguity.

Q: Why Person 2's Response + ResponseBuilder instead of Person 3's Respond?
A: Respond is monolithic (8 flat args, mixed logic + I/O). Response + ResponseBuilder
   cleanly separates data from serialization. Easier to test and extend.

Q: Why keep Person 3's config parsing?
A: Parser works end-to-end and is not replaced by Person 2's code. Wrapping it with
   the adapter preserves all existing config functionality.

Q: Why rewrite Router?
A: Person 3's Router took Request&; Person 2's HTTPRequest is const-safe.
   Also dropped get_path/get_error_path—MethodHandler already implements this logic,
   keeping two copies would cause maintenance bugs.

Q: Why RouteConfig adapter?
A: Person 2's MethodHandler already calls route.getX() methods (looked for at merge
   time). Those methods were never implemented. The adapter realizes the contract
   that was already written into MethodHandler's code. Format mismatches handled:
   - Methods map → vector
   - Redirect map → single tuple

Q: Why stub CGI?
A: Person 1 hasn't implemented CGIHandler yet. Guarding with #ifdef HAVE_CGI allows
   the codebase to compile without CGI; Person 1 flips the flag and implements the
   interface. Clean handoff contract.
