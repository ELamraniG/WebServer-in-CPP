## What Was Done

Merged three codebases into a unified `webserv/` project:

1. **Person 3** — Config parsing (`Tokenizer` → `Parser` → `server_block` + `location_block`)
2. **Person 2** — HTTP request/response (`HTTPRequest` → `RequestParser` → `MethodHandler` → `ResponseBuilder`)
3. **New** — `RouteConfig` adapter (bridges the two)

## Key Decisions

### 1. Request Type Winner: HTTPRequest (Person 2)
- Proper class with getters/setters
- Lowercased header keys (matching HTTP/1.1 semantics)
- Chunked decoding built in
- Person 3's stub was explicitly marked "NOT my task"

### 2. Response Type Winner: Response + ResponseBuilder (Person 2)
- Clean separation: data struct vs. serialization
- Person 3's monolithic `Respond::generate_response(8 args)` mixed concerns

### 3. Config Type Winner: server_block + location_block (Person 3)
- Working parser that consumes Person 3's config files
- No rewrite needed; wrapped with adapter

### 4. New: RouteConfig Adapter
**Most critical piece.** Person 2's code already called `route.getX()` methods everywhere but the interface was never implemented. RouteConfig does:

- Takes `(const server_block&, const location_block*)`
- Flattens `map<string,bool>` methods → `vector<string>` (only true keys)
- Collapses `map<int,string>` redirects → single `(code, url)` pair
- Exposes: `getRoot`, `getIndex`, `getAutoindex`, `getAllowedMethods`, `getRedirect`, `getRedirectCode`, `getUploadStore`, `getErrorPages`, `getMaxBodySize`, `getCgiPass`, `getLocationPath`
